Git Cheatsheet

By: Andrew 'Azure-Agst' Augustine

How do I save my work?

Remember the holy trinity of commands! Make sure you're on a feature or bug branch, and run these:

$ git add *
$ git commit -m "Summary of what you did"
$ git push

The git add command tells git to add a file to be staged for the next commit. (Note: Similarly, if you want a file to be removed from git, you can use git rm)

The git commit command tells git to take a shapshot of the changed you've made to your staged files, and save them as a commit.

The git push command takes the commits you've made, and uploads them to github.

How do I initially set up my development environment?

There are a few commands that we'll need to run to get the repo into shape.

First, clone the repo and cd into it:

$ git clone https://github.com/[yourUsernameHere]/CreatorConnect
$ cd CreatorConnect

Now, make sure you're looking at the develop branch.

$ git checkout develop

So, we have your fork of the repo, but we need to make sure it's in sync with the Hub's repo. To do that, we'll need to set an upstream.

Run this command on your computer and check if your response looks like this:

$ git remote -v
origin https://github.com/Azure-Agst/CreatorConnect (fetch)
origin https://github.com/Azure-Agst/CreatorConnect (push)
upstream https://github.com/FSUInnovationHub/CreatorConnect.git (fetch)
upstream https://github.com/FSUInnovationHub/CreatorConnect.git (push)

If you're missing the two upstream entries, run this command:

$ git remote add upstream https://github.com/FSUInnovationHub/CreatorConnect.git

Now, whwnever you want to sync with the Hub's version of develop, run this:

$ git pull upstream develop

How do I go about making a new branch?

We do not allow people to push to the FSUInnovationHub's master or develop branches directly. We only allow pull requests from feature branches or bug branches to be merged into develop, and live software to be pushed to master.

So how do you name a new branch? Which should you use?

  • If you're working on a feature, use a feature branch!
    Feature branches are prefixed with f/ and are usually one word the describes what you're working on. (I.e. f/newFeature)
  • If you're working on a bug fix, use a bug branch!
    Bug branches are prefixed with b/ and are numerically assigned in ascending order. (I.e. The 123rd bug fix would be on branch b/123)

To make a new branch, make sure you're already in the develop branch of your fork, and use this:

$ git checkout -b [newBranchName]

This creates a new branch on your local machine. Make sure your upstream is still set by doing git remote.

$ git remote -v

Once you're on your new branch, make your changes! Use git add and git commit to save your changes to a commit.

Once you're ready to push, follow the instructions in the following question.

My current branch has no upstream? What does that mean?

Now, you can't just git push on a new branch because git checkout -b doesn't automatically make a corresponding branch on GitHub for us! To tell GitHub to make a new branch for us, we use this command:

$ git push origin [newBranchName]

Github should now make that corresponding branch for us and upload the code from our machine into it. Now, use the website to make a pull request, and merge into develop!

How do I check what files are staged for my next commit?

You can see what files are staged for addition, deletion, modification, or are ignored in your next commit by running this command:

$ git status

Develop has updated cince I made my feature branch! What do I do?

Don't worry! When you merge your feature branch's commits into develop, it should automatically take care of any conflicts for you.
